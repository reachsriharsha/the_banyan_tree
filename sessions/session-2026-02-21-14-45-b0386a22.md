# SDLC Plan

Structured ideation skill that takes a raw idea and produces a clear, finalized Decision Summary ready for spec creation.

## When This Skill Activates

Trigger phrases:
- "let's brainstorm [feature/idea]"
- "I have an idea for..."
- "what if we built..."
- "help me think through..."
- "I want to explore..."
- "let's plan [feature]"
- "new feature idea"

## The Planning Process

### Phase 1: Idea Capture (1-2 messages)

When the user presents an idea, do NOT immediately ask 10 questions. Instead:

1. Restate the idea in one clear sentence to confirm understanding
2. Ask the single most important clarifying question (usually: "Who is this for and what problem does it solve for them?")
3. Wait for response before proceeding

### Phase 2: Structured Exploration (3-5 messages)

Work through these areas conversationally — do not dump all at once:

**Problem clarity**
- What is the user pain point this solves?
- How does this fit within the project's existing direction?
- What happens if we don't build this?

**Scope definition**
- What is the simplest version of this (MVP)?
- What would a full version look like?
- What is explicitly out of scope?

**Technical considerations**
- What parts of the existing system does this touch?
- Are there dependencies or blockers?
- Any technical risks worth flagging early?

**User journey**
- Walk through the feature from the user's perspective step by step
- Identify edge cases and unhappy paths
- Confirm the primary success scenario

### Phase 3: Decision Summary (final output)

When the idea is sufficiently explored, produce a Decision Summary in this exact format:

```
## Decision Summary

**Feature Name:** [Short, clear name]
**Date:** [Today's date]
**Status:** Ready for Spec

### Problem Statement
[2-3 sentences. What problem this solves and for whom.]

### Proposed Solution

### MVP Scope

### Out of Scope (v1)
[Bulleted list of what we are explicitly NOT building now]

### Key Design Decisions
[Any important decisions made during brainstorm that should inform the spec]

### Open Questions
[Any unresolved questions that need to be answered during spec or design]

### Next Step
Run sdlc-spec on this Decision Summary to generate the Feature Spec.
```

## Rules

- Never write a Decision Summary until Phase 2 is complete
- Never start writing spec content during brainstorm — that belongs to sdlc-spec
- If the user says "let's spec this out" or "I'm happy with this", immediately produce the Decision Summary and tell them to run sdlc-spec next
- Keep the tone conversational and collaborative, not interrogative
- If the user wants to skip structure and just think out loud, match their energy — formalize at the end

## Error Handling

**If the idea is too vague to make progress:**

**If the user jumps to solutions before the problem is clear:**
Say: "Before we lock in the solution, let's make sure we agree on the problem — that'll keep the spec tight. Who is experiencing this and what are they trying to do?"

**If scope keeps expanding:**

# SDLC Spec

Generates complete, structured Feature Specs for software features.

## When This Skill Activates

Trigger phrases:
- "spec this out"
- "create a spec for..."

## Pre-Flight Check

Before writing ANY spec content:

1. **Check for a project-specific template** in `references/` — if one exists, read it first and follow its conventions
2. **Confirm input quality** — do you have enough context? You need at minimum:
   - A problem statement
   - A proposed solution
   - Rough scope (what's in, what's out)
3. If input is insufficient, ask: "Before I write the spec, I need a bit more clarity on [specific gap]. Can you help me fill that in?"

## Feature Spec Structure

Generate the Feature Spec document in a single response:

```

**Date:** [Today's date]
**Author:** [Ask user or leave as TBD]
**Status:** Draft

---

## Overview
[2-3 sentence summary of the feature]

## Problem Statement
[Clear description of the user problem. Include who is affected and the impact.]

## Goals
[Numbered list of specific, measurable outcomes this feature achieves]

## Non-Goals
[Explicit list of what this feature does NOT do — critical for preventing scope creep]

## User Stories
[Format: As a [user type], I want to [action] so that [benefit]]
[Include primary flow + 2-3 edge case stories]

## Functional Requirements
[Numbered list. Each requirement should be testable.]
FR-001: [Requirement]
FR-002: [Requirement]
...

## Acceptance Criteria
[For each FR, what does "done" look like?]
AC-001: [Criteria for FR-001]
...

## Success Metrics
[How will we know this feature is working? What do we measure?]

## Dependencies
[Other features, systems, or teams this depends on]

## Open Questions
[Unresolved questions that need answers before or during development]
```

## After Generating the Spec

End with this block:

```
---
## Spec Complete

**Next steps:**
1. Review and fill in any [TBD] or [placeholder] sections

3. Run sdlc-test to generate the Test Spec

```

## Rules

- Use FR-001, AC-001 numbering consistently — it makes cross-referencing easier across design and test specs
- If something is genuinely unknown, write [TBD — needs input from X] rather than guessing
- Keep Feature Spec in product language (no code) — technical details belong in sdlc-design
- If the input came from sdlc-plan, pull the Open Questions into the spec directly
- Always generate a complete spec in one response unless the user asks otherwise

Say: "This is a large feature. I'd recommend we split it into [X] and [Y] and spec them separately. Want me to start with [X]?"

**If there are conflicting requirements:**
Flag them explicitly in the Open Questions section rather than silently resolving them.

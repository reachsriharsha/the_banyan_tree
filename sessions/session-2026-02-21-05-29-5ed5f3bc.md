## The problematic path: trip creation after server restart

After a server restart, all singletons start fresh. Consider this sequence:

1. **Server starts** → `glob_trip_manager = TripManager()` → `_fcm_manager = None`
2. **Driver creates a trip** → `POST /api/groups/trip/create`
3. `routers/trips.py` calls `glob_trip_manager.create_active_trip()`
4. `trip_manager.py:147` accesses `self.fcm_manager` (the property)
5. The lazy property at `trip_manager.py:42-46` sees `self._fcm_manager is None` → **creates its own `FCMManager()`**
6. **Later**, a group update happens → `routers/groups.py:49` calls `_get_fcm_manager()`
7. `_get_fcm_manager()` creates a **second** `FCMManager()` and overwrites `glob_trip_manager._fcm_manager`

**Result:** Two FCMManagers created, and the first one (from step 5) is orphaned — its ThreadPoolExecutor is never shut down, never garbage collected (the property returned a reference that was used by the thread pool's internal threads).

## Why this is likely in practice

A trip can be created without any group membership change in the same server session. The driver and group can be set up in a previous server lifecycle. After a restart:

- The first API call that touches `glob_trip_manager.fcm_manager` will be the one that wins
- `_get_fcm_manager()` is only called from `routers/groups.py:_send_group_refresh_notification` — it's only triggered by group membership/driver changes
- `glob_trip_manager.fcm_manager` is triggered by trip create/update/finish — which is the most common operation

So in practice, **trip operations will almost always hit the lazy fallback first**, making the injection in `_get_fcm_manager()` arrive too late.

## The fix

The injection should happen eagerly in `globals.py` rather than lazily inside `_get_fcm_manager()`. Two options:

1. **Create FCMManager eagerly in `globals.py`** and pass it to both `glob_trip_manager` and `_get_fcm_manager`:
   ```python
   _fcm_manager = FCMManager()
   glob_trip_manager = tm.TripManager(fcm_manager=_fcm_manager)
   ```

2. **Have TripManager call `_get_fcm_manager()`** in its property — but this creates the circular import we identified earlier.

Option 1 is the clean solution. It does mean FCMManager is created at import time (not lazily), but since `globals.py` is only imported during FastAPI startup, that's fine. Want me to make this change?
